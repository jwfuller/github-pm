Project Management using GitHub

This is a brief introduction to using GitHub for Project Management. I want to give you a tour through major features and show you a little of what is possible. This is not intended as a training or to be comprehensive.

GitHub Structures

  • User
  • Organization
  • Repository (repo)
  • Issue
  • Discussion
  • Project
  • Everyone signs into GitHub with a Personal Account.
  • Personal accounts are then associated with our Organization Account.
  • A Repository is a location in GitHub where files, code, and Issues are stored and managed.
  • Repositories can be owned by either a Personal or Organization Account.
  • Each Repository has option to enable the Issues and Discussions features which enables you to track tasks and bugs, and discuss ideas or solicit feedback.
  • You can use Projects to organize and prioritize Issues.

Repositories

A repository contains all of your project's files and each file's revision history. It can be public or private.

Issues, Labels, and Milestones

All of these exist within a repository and are specific to the repository they are created in.
A list of Open GitHub issues on the SCAG project
Can filter the list by Author, Label, Project, Milestone, and Assignee. All of these filters are represented in the Filter search box and the URL, so it is easy to power-user search and share a specific view like "All open issues, labeled with Bug, and in Milestone PI 23.1.4". The filter syntax is the same in other places in GitHub like Projects.

Here is an Issue, this one is using the Checklist feature.
The first comment in an Issue is special, it serves as the Description. Checklist items in the field are aggregated to a # of x tasks at the top of the issue. Some of the checklist items have been turned into their own Issues. You can see the status as well via the green dot or purple check. There is also a nifty feature to convert items to Issues, so you can write out a list of tasks and then turn them into full Issues later.
Tasks can be assigned to up to 10 people
Default labels are assigned when a repo is created, and can be customized on a per repo basis. If the default labels are changed at a later date, repos are not updated.
Issues can be assigned to on or more projects, and you can manage the per-project status here as well.
An issue can be placed into a Milestone. Milestones exist at the repo level and are not shared across repos.
You can follow Issues, Repositories, or Organizations. There are also different levels of notifications which I will touch on later.
Labels and Milestones are accessed from the Issue List page.
Labels allow you to cross-cut your issue list. You can also see a "Convert Issues" button, that allows you to convert all issues with a specific label into a Discussion.
Milestones can be assigned Due Dates, have a description, and show progress towards completion.

Discussions

Use discussions to ask and answer questions, share information, make announcements, and conduct or participate in a conversation about a repository. There are different type of Discussions including Announcements (only maintainers can create), polls, Q&A (allows the best answer to be voted on), and open ended discussion.
You can also lock discussions if they become heated or come to a conclusion. Discussions use the same labels as Issues and can be converted to Issues.

Projects and Views

Board View

Projects exist at the Organization level, so can be used across repositories. This is a Board view of a project with Columns created from a custom field, in this case for Status.

Changing Views

You can create up to 42 views per project. Each view can have it's own display characteristics and saved filters. There are also great filters like `assignee:@me` that allows you to create one "My Work" view that works for all users. The small arrow next to a View name gives you access to the customization options for the view.

Custom Project Fields

  • Text
  • Number
  • Date
  • Single select list
  • Iteration
Several different types of custom fields. 'Iteration' allows you to setup repeating time blocks (like 2 week sprints). You can filter for the 'current' or 'next' iteration. These fields are specific to the Project, so while they feel a lot like Milestones, there is not a connection between the two structures. There are some other cool things you can do like summing a 'points' field and showing that at the top of Board with Columns for Iteration

Filtering Issues

You can filtering Issues by things like Assignee, Label, Milestone, Iteration, Status. These filters are used Project boards and in the full list of issues for a Repository. And like with Repos, the URL reflects your filters so it is easy to share what you are looking at.
